# task-plan-mode — 任务规划模式

## 一、概述

**为什么需要 Plan Mode？**

复杂任务的常见失败模式：
- **上下文溢出**：长对话丢失早期信息
- **方向偏差**：做到一半发现不是用户要的
- **信息不全**：用户给的条件不够做出好结果

**Plan Mode 的核心思想：**
1. Agent 主动判断任务复杂度，决定是否进入规划模式
2. 在记忆区创建独立 workspace，逐步收集需求
3. 每次只推进 1 个环节，根据信息完整度自行判断
4. 信息足够时自动退出，直接进入执行

## 二、触发规则

### 自动触发（默认）

| 触发条件 | 判断方式 | 示例场景 |
|---------|---------|---------|
| 任务表述包含"做一个X" | 用户说出做网站/开发/设计等 | "帮我做一个二手书交易小程序" |
| 任务涉及 >3 个步骤 | agent 内部评估 | 写深度报告、搭全栈项目 |
| 用户提到模糊需求 | 需求不明确、缺关键信息 | "帮我做个东西，关于新能源的" |
| 用户说"先规划一下" | 明确要求先拆解 | "你先规划一下这个任务" |

### 手动触发

以下关键词直接进入 Plan Mode：
- "切任务模式" / "Plan Mode"
- "帮我拆解一下"
- "这个任务比较复杂"

### 触发回复模板

```
✅ 已切换为任务规划模式（Plan Mode）

第一步：{[第一步问题]}
（信息够了我会自动退出规划模式，进入执行）
```

### 目录结构

```

├── plan.md            # 拆解计划
├── context/           # 需求收集记录
│   ├── 01-需求对齐.md
│   ├── 02-功能确认.md
│   └── 03-技术方案.md
```

### 初始化动作

1. 创建目录
2. 写入 README.md（目标、范围、状态 = planning）
3. 写入 plan.md（空阶段表，逐步填充）

## 四、需求对齐流程

### 核心原则

**Agent 是流程专家，用户是需求方。**
用户不知道做一个小程序需要准备哪些东西——这是正常的。Agent 的职责是按领域知识给出结构化的需求清单，每次只推进 1 个环节。

### 通用提问框架

1. 最终交付物是什么？
2. 主要目标是什么？
3. 有没有截止时间？
4. 有没有参考案例或竞品？
5. 有没有已有的素材或数据？

### 推进逻辑

```
Step 1: 问 1 个问题 → 用户回答 → 写入 context/
Step 2: 根据回答决定下一轮问题 → 再问 1 个 → 用户回答 → 写入
Step 3: 重复，直到 agent 判断信息足够
Step 4: 信息足够 → 自动退出 Plan Mode → 生成完整 plan.md
```

## 五、领域脚手架

### 网站 / 小程序开发

| 环节 | 需求收集 | 典型问题 |
|------|---------|---------|
| 1 | 目标用户 | 谁用？C端还是B端？ |
| 2 | 核心功能 | 最重要的 3 个功能？ |
| 3 | 技术选型 | 有偏好的技术栈吗？ |
| 4 | 页面结构 | 需要几个页面？ |
| 5 | 数据需求 | 用户系统？后台管理？ |
| 6 | 设计风格 | 有参考网站或品牌色？ |

### 文章 / 报告写作

| 环节 | 需求收集 | 典型问题 |
|------|---------|---------|
| 1 | 目标读者 | 写给谁看？ |
| 2 | 核心论点 | 要传达的核心观点？ |
| 3 | 篇幅要求 | 多少字？ |
| 4 | 数据来源 | 需要引用数据吗？ |
| 5 | 风格要求 | 正式/活泼/学术/极简？ |

### 研究 / 分析

| 环节 | 需求收集 | 典型问题 |
|------|---------|---------|
| 1 | 研究问题 | 要解决什么具体问题？ |
| 2 | 范围界定 | 行业/时间/地域范围 |
| 3 | 数据需求 | 需要哪些类型的数据？ |
| 4 | 分析方法 | 定性/定量/对比/趋势？ |
| 5 | 输出格式 | 报告/PPT/数据看板？ |

### 其他领域

不属于以上三类 → agent 自动生成领域专用脚手架：
- 从用户描述中推断任务类型
- 列出该领域的关键决策环节
- 按环节逐步提问

## 六、出口判定（自动退出）

### 退出条件

每次用户回答后自我评估：

```
信息充分度 checklist：
☐ 任务目标已明确且有具体产出描述
☐ 核心需求已覆盖
☐ 没有明显的关键信息缺口
☐ 可以开始执行
```

**全部打勾 → 自动退出。** 不再询问。

### 退出通知模板

```
✅ 信息收集完毕，退出规划模式。

任务总览：{一句话}
核心需求：{3-5 条要点}

进入执行阶段。
```

### 超时处理

- 同一环节最多追问 2 轮
- 用户反复给不出答案 → 默认建议 + 标注"用户未确认"

---

## 七、退出后：执行阶段

1. 每个阶段产出一个交付物放入 deliverables/
2. 每阶段完成后主动汇报
3. 需要重新决策时短暂回到需求对齐

### 进度汇报模板

```
[阶段 2/5 完成] 核心功能设计
✅ 已产出：功能清单与优先级
🔜 下一阶段：技术选型与架构设计
```

## 八、完整示例

### 场景：做二手书交易小程序

```
Step 1 — Agent 提问：
✅ 已切换为任务规划模式
第一步：这个二手书小程序主要是给谁用的？
→ 用户：广西大学的学生

Step 2 — Agent 继续：
最核心的 3 个功能是什么？
→ 用户：发布和浏览最重要

Step 3 — Agent 继续：
需要用户注册登录吗？微信授权？
→ 用户：微信授权

Step 4 — Agent 自动评估：信息足够
✅ 退出规划模式

任务总览：广西大学二手书交易小程序
核心需求：微信登录 → 发布 → 浏览/搜索

进入执行阶段。
```
