# task-plan-mode — 任务规划模式 v2.0 🎯

> **让你的 AI Agent 拥有分步式任务拆解+执行能力。**
>
> 复杂任务不再是"一轮对话就产出"。AI 先做前置评估 → 明确分几个阶段 → 每个阶段逐步执行 → 阶段间汇报进度。

---

## 一、核心流程

```
用户提出任务
       ↓
【Phase 0】前置评估 —— AI 输出任务评估与阶段划分
       ↓
【Phase 1】需求确认 —— 按领域脚手架逐步收集需求
       ↓
【Phase 2-4】分步执行 —— 按评估阶段逐一产出
       ↓
【Phase Final】汇总交付
```

**关键原则：** 每一步完成后汇报进度，用户确认后才进入下一步。

---

## 二、触发规则

### 自动触发

| 触发条件 | 判断方式 | 示例 |
|---------|---------|------|
| 任务涉及 >2 个步骤 | agent 内部评估 | 做网站、写报告、开发项目 |
| 模糊需求、缺关键信息 | 需求不明确 | "帮我做一个关于新能源的东西" |
| 包含"做/开发/设计/写"等动词 + 复杂对象 | 任务复杂度高 | "做一个二手书交易小程序" |

### 手动触发

关键词：`规划模式` / `Plan Mode` / `帮我拆解` / `先规划一下`

---

## 三、Phase 0：前置评估（新增）

这是本次升级的核心。收到任务后，AI **不是直接问需求**，而是先输出一份任务评估：

### 评估模板

```
📋 任务评估

收到任务：[用户原始需求]

我计划将本任务分为以下 X 个阶段完成：

Phase 1：需求确认
  → 收集核心需求、目标用户、技术选型

Phase 2：[如"前端设计"]
  → 页面结构、交互流程

Phase 3：[如"后端开发"]
  → 数据模型、API 接口

Phase 4：[如"联调与交付"]
  → 整合测试、最终交付

当前进度：Phase 0 — 评估完成
下一步：开始 Phase 1 需求确认，请确认是否按此方案进行？
```

### 评估判断规则

AI 根据任务内容自动判断阶段划分：

| 任务类型 | 默认阶段划分 |
|---------|------------|
| 网站/应用开发 | 需求确认 → 前端 → 后端 → 联调交付 |
| 文章/报告写作 | 需求确认 → 大纲 → 正文 → 排版交付 |
| 研究分析 | 需求确认 → 数据搜集 → 分析 → 报告 |
| 设计类 | 需求确认 → 概念设计 → 细化 → 交付 |
| 自定义 | AI 动态生成 |

---

## 四、Phase 1：需求确认

用户确认阶段划分后进入需求收集。

### 推进规则

```
Agent 问 1 个问题 → 用户回答 → 写入 workspace
            ↓
判断信息是否足够？ → 不足 → 再问 1 个
            ↓
         足够 → 进入 Phase 2
```

### 领域脚手架

#### 网站/小程序开发

| 轮次 | 需求维度 |
|------|---------|
| 1 | 目标用户与场景 |
| 2 | 核心功能（最重要的 3 个） |
| 3 | 技术栈偏好 |
| 4 | 页面/模块结构 |
| 5 | 设计风格参考 |

#### 文章/报告

| 轮次 | 需求维度 |
|------|---------|
| 1 | 目标读者与核心观点 |
| 2 | 篇幅与风格 |
| 3 | 数据来源要求 |

#### 研究分析

| 轮次 | 需求维度 |
|------|---------|
| 1 | 研究问题与范围 |
| 2 | 数据类型与来源 |
| 3 | 分析方法与输出格式 |

### 每轮最多追问 2 次

用户给不出答案 → 默认建议 + 标注"用户未确认"

---

## 五、Phase 2+：分步执行

需求确认完成后，按评估阶段逐一执行。

### 执行规则

- **单线程：** 一次只做一个阶段
- **阶段汇报：** 每阶段完成后主动汇报

### 汇报模板

```
✅ Phase [N/M] 完成：[阶段名]

已完成：{具体产出摘要}
🔜 下一步：Phase [N+1] — [下一阶段名]

是否有需要调整的地方？确认后继续。
```

### 阶段切换

用户确认后进入下一阶段。如果有调整，回到对应阶段修改。

## 六、Workspace 结构

```
~/.task-plan-mode/workspace/{项目名}/
├── README.md          # 项目总览（含评估结论）
├── plan.md            # 阶段计划（含完成状态）
├── phase-0-assessment.md  # 前置评估记录
├── phase-1-requirements/  # 需求收集
├── phase-2-output/        # 阶段 2 产出
├── phase-3-output/        # 阶段 3 产出
└── deliverables/          # 最终交付物
```

## 七、信息流与状态同步

### 当前进度在哪儿？

agent 每次回复时在结尾标注：

```
📌 当前：Phase 2/4（前端设计）— 页面结构已确定，下一步交互流程
```

### 阶段性成果记录

每个阶段产出一个存档文件，记录决策理由。

---

## 八、退出判断

### 正常退出
所有阶段完成 → 汇总交付。

### 提前退出
- 用户说"不用规划了，直接做" → 跳过评估，直接执行
- 用户说"先做一部分看看" → 只做 Phase 1-2
- 用户 24 小时无回复 → 暂停规划，标注"等待用户确认"

## 九、完整示例

### 场景：做一个新能源行业分析报告

```
用户：帮我写一份关于新能源行业趋势的分析报告

AI（Phase 0 — 前置评估）：
📋 任务评估
我计划将本任务分为以下 4 个阶段：
Phase 1：需求确认
 Phase 2：数据搜集（行业数据、政策、技术趋势）
Phase 3：分析撰写（分章节产出）
Phase 4：排版交付（图表+最终稿）
当前：Phase 0 完成。请确认是否按此方案执行？

用户：可以

AI（Phase 1 — 需求确认）：
第一步：这份报告的目标读者是谁？公司内部决策层？行业媒体？还是公众号读者？

用户：公众号读者，3000-5000字

AI：核心要传达的观点是什么？看多新能源？还是客观分析各技术路线？

用户：客观分析，但重点突出固态电池和氢能的商业化进展

AI：需要引用数据吗？我可以帮你搜集。

用户：需要，近一年的行业数据最好

（信息足够 → 自动进入 Phase 2）

📌 当前：Phase 2/4（数据搜集）
...
```
